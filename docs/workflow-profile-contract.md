# Workflow Profile Contract

Generated by /plan-eng-review on 2026-07-02
Branch: main
Repo: sonim1/preqstation
Status: DRAFT

## Problem

PreqStation needs to record how a harness chose to run task work without making PreqStation Core responsible for choosing gstack, superpowers, Speckit, OpenSpec, or any other workflow.

The default user experience should be hands-off: the user dispatches work, the harness decides the right workflow, and Core stores the resulting graph memory.

## Locked Decisions

1. Use a contract-first slice. Do not build the full manual selector yet.
2. Default workflow profile is `auto`.
3. PreqStation Core does not choose or execute concrete skills.
4. Harnesses own semantic judgment and may choose the actual workflow/tool.
5. Store requested and resolved workflow information in existing work-node `metadata`.
6. Use a namespaced metadata object: `metadata.workflow_profile`.
7. Do not append `workflow_profile="auto"` to current Telegram/Hermes dispatch commands yet.
8. Keep existing task notes attached only to the canonical root node.
9. New work nodes are created dynamically while the runtime proceeds.

## Existing Code

- `lib/work-graph-service.ts` already accepts and persists `CreateWorkNodeInput.metadata`.
- `app/api/tasks/[id]/work-graph/nodes/route.ts` already accepts `metadata` and passes it to `createWorkNode`.
- `preqstation graph node create` and node transition commands accept `--metadata-file` so CLI-based harnesses can pass the same metadata contract.
- `preqstation agent guide --json` exposes the workflow profile contract so harnesses can discover the default `auto` profile and resolved metadata fields.
- `preq_agent_guide` exposes the same workflow profile contract to MCP-based harnesses.
- `lib/openclaw-command.ts` already supports command metadata, but this slice does not add `workflow_profile` to dispatch messages.
- `app/components/work-graph-tree.tsx` and existing tests already keep root notes on the canonical root node.
- `app/components/work-graph-node-inspector.tsx` shows stored workflow profile metadata for selected work nodes.

## Contract

When no manual workflow override exists, the requested profile is:

```json
{
  "workflow_profile": {
    "requested": "auto",
    "manual_command": null
  }
}
```

After the harness chooses and runs a workflow, it should create or update the relevant work node with:

```json
{
  "workflow_profile": {
    "requested": "auto",
    "manual_command": null,
    "resolved": "gstack-plan-eng-review",
    "resolved_command": "/plan-eng-review",
    "resolved_reason": "Architecture review needed before implementation"
  }
}
```

Field rules:

- `requested`: required string. For this slice, only `auto` is required.
- `manual_command`: nullable string. Must be `null` unless a later manual override UI exists.
- `resolved`: optional string chosen by the harness.
- `resolved_command`: optional string showing the concrete workflow command or tool.
- `resolved_reason`: optional short human-readable reason.

## Data Flow

```text
User dispatches task
  |
  v
PreqStation sends existing dispatch command
  |
  v
Harness reads task + graph + local repo context
  |
  v
Harness chooses workflow because requested profile is auto
  |
  v
Harness creates/updates work node
  |
  |  API/MCP direct body or CLI --metadata-file
  |
  v
Core stores metadata.workflow_profile on that node
  |
  v
Task graph preserves process, evidence, and workflow choice
```

## Failure Modes

- Harness does not write `metadata.workflow_profile`: graph still works, but the workflow choice is not preserved.
- Harness writes flat keys instead of the namespace: data is harder to distinguish from unrelated metadata.
- A future implementation appends `workflow_profile` to dispatch commands before parser compatibility is verified: strict external harness parsers may reject the command.
- Future UI treats manual workflow as default: PreqStation starts looking like the semantic decision-maker, which conflicts with the runtime boundary.

## Not In Scope

- Manual workflow selector UI.
- Project-level default workflow settings.
- CLI flags for workflow profile selection.
- MCP tool schema expansion for manual workflow profile selection.
- Dedicated DB columns or indexes for workflow profile fields.
- Appending `workflow_profile` to existing dispatch command metadata.
- Public API or guide documentation updates.
- Choosing or executing gstack, superpowers, Speckit, OpenSpec, or any other concrete skill inside Core.

## Parallelization

Sequential implementation, no parallelization opportunity. The slice is small and centered on one contract path.
